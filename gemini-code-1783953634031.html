Coding Lab: Verifying Live Reload Behavior

In this practical lab session, we open the IDE to implement automatic reloading, inject alternative routing points, and watch the console refresh automatically.

Now that our development ecosystem is configured, let's verify how Spring Boot DevTools cuts out manual compilation loops. We will run our primary application class and introduce dual mock REST endpoints while keeping our server runtime completely active.

Step-by-Step Lab Execution Pipeline
  • 1 Initialize the Runtime: Open your main entry class file MyAppApplication.java and launch the application. Ensure the integrated shell console boots completely on port 8080.
  • 2 Append Endpoints: Open your custom SimpleRestController.java source class to expand your system behaviors with the live mock mappings below.
Updated Rest Controller Blueprint

Add these two continuous mapping methods directly inside your controller body without stopping or terminating your active IDE console execution:

@RestController public class SimpleRestController { // Existing base endpoints match here... // Expose new endpoint for learning goals @GetMapping("/learning") public String getLearningGoals() { return "Learn 4 modules each day."; } // Expose new endpoint for gaming levels @GetMapping("/gaming") public String getDailyLevels() { return "Complete 10 levels per day."; } }
Observing the Console Activity Stream

When you press Ctrl+S (or Cmd+S on macOS) to save your code adjustments, look closely at your lower terminal console. You will notice the log screen instantly scroll as DevTools drops the classloader and re-allocates endpoints without taking down your underlying process.

Verifying Output inside the Browser

Open your local web browser tool to test both dynamic targets. The system loads your new business logic immediately without requiring any manual server restarts:

http://localhost:8080/gaming
